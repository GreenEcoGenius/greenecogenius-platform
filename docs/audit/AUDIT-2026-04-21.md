## P0, à corriger en priorité (bloquant qualité / prod)

### 1. Erreur TypeScript en production build

Fichier : [`apps/web/app/[locale]/home/(user)/rse/page.tsx:37`](../../apps/web/app/[locale]/home/(user)/rse/page.tsx#L37)

```
error TS2345: Argument of type 'Translator<Record<string, any>, "rse">' is not assignable to parameter of type 'TFn'.
```

Bloque `pnpm typecheck` proprement. Probablement une fonction qui reçoit `TFn` strict mais accepte un translator `Record<string, any>`. Fix : soit adapter le type d'entrée de la fonction consommatrice, soit caster proprement via `as unknown as TFn` avec commentaire, soit changer la signature du consommateur pour accepter un translator permissif.

### 2. Billing webhook sans vérification de signature

Fichier : [`apps/web/app/api/billing/webhook/route.ts`](../../apps/web/app/api/billing/webhook/route.ts)

Le handler utilise `enhanceRouteHandler` + `getBillingEventHandlerService` qui déléguent probablement la vérif au `@kit/billing-gateway`. **À vérifier** que le provider Stripe vérifie bien `constructEvent(body, sig, webhookSecret)` dans ce chemin, sinon n'importe qui peut poster un event fake.

Les 4 autres webhooks sont OK :
- `stripe/webhook` : `constructEvent` présent
- `stripe/connect/webhook` : `constructEvent` présent
- `webhooks/docusign` : vérif signature présente
- `db/webhook` : `dbWebhookSecret` vérifié

Action : inspecter [`@kit/billing-gateway`](../../packages/billing/gateway/src) pour confirmer la vérif. Si absente, bug sécu critique.

## P3, sécurité (revue)

### ✅ Bonnes pratiques confirmées

1. **Middleware `proxy.ts`** protège correctement `/admin/*` (super admin required), `/home/*` (auth required), `/auth/*` (redirect si déjà connecté). Les 3 pages dashboard qui font `getSupabaseServerClient()` sans `requireUser()` explicite (`marketplace/[id]/confirmation`, `marketplace`, `pricing`) sont en fait protégées par le middleware + héritent RLS.

2. **Admin client usage** : 13 occurrences de `getSupabaseServerAdminClient()` trouvées, toutes dans des routes légitimes (server actions `transactions`, `external-activities`, `billing`, routes `join/accept`, `verify`, `api/flux/generate`, `api/verify`, `pricing/page`). Bypass RLS justifié dans chaque cas (lecture plans publics, génération PDF, acceptation invitation, etc.). **À auditer manuellement** : chaque endpoint admin doit valider l'identité utilisateur avant de lire/écrire des données user-owned.

3. **Cron route** `api/cron/update-materials` vérifie `CRON_SECRET` via `Bearer` header. ✅

4. **Contact form** : `publicActionClient` + `inputSchema` Zod + `escapeHtml()` avant envoi email. ✅ Pas de XSS.

5. **`dangerouslySetInnerHTML`** : 6 usages, tous sur du contenu CMS contrôlé (blog description, docs-card subtitle, faq JSON-LD structured data, changelog description, verify script dom). **À vérifier** : si `CMS_CLIENT=wordpress` ou autre CMS externe, le contenu est-il assaini avant stockage ? Si Keystatic/Markdoc (local), c'est safe.

6. **Pas de secret serveur leaké** dans les components. Les `NEXT_PUBLIC_*` exposés sont bien publics par design (style nav config).

### ⚠️ Points d'attention

1. **Cookies sans bannière** : 6 usages de `document.cookie` directement (NEXT_LOCALE, last-selected-team). Techniquement ces cookies fonctionnels ne nécessitent pas consent RGPD explicite (finalité légitime, pas de tracking). Mais vérifier que le cookie-banner de `@kit/ui` gère bien ces cas si tu actives analytics.

2. **Admin actions sans vérif super-admin explicite** : pas de `isSuperAdmin()` check dans `apps/web/app/api/` routes. Le middleware bloque `/admin/*` mais les routes `/api/admin/*` (si elles existent) doivent vérifier manuellement. Grep n'a rien trouvé, donc pas d'API admin exposée, ✅.

3. **Pas de rate limiting visible** sur les server actions publiques (contact form). Un attaquant peut spam le endpoint `sendContactEmail`. À envisager : Arcjet, Turnstile, ou `@upstash/ratelimit`.

4. **Cookies avec `SameSite=Lax`** : OK pour la plupart des cas, mais pour `NEXT_LOCALE` et cookies de sélection team, `Lax` expose à certains scénarios CSRF. Moins critique car pas auth-sensitive.

5. **7 vulns dépendances high** (voir P1). Dont 1 `next.js` DoS et 1 `lodash` code injection potentiellement exploitables.
